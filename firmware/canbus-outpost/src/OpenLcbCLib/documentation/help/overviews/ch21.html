Chapter 21 — Multi-Node Message Routing

A single physical device can host multiple virtual OpenLCB nodes — for example, a command station might host one configuration node plus several train nodes. Each virtual node has its own unique 48-bit Node ID and its own 12-bit CAN alias. From the perspective of other devices on the CAN bus, each virtual node is an independent node.

Implementation status: Sibling dispatch is fully implemented in openlcb_main_statemachine.c and openlcb_login_statemachine.c, covering all three phases: incoming message fan-out, outgoing sibling dispatch, and login-time sibling notification. See also Ch 8 §8.6 for the dispatch mechanism details.

21.2 The Loopback Problem

When virtual Node A transmits a message, the CAN hardware does not deliver that frame back to the local receive path. Remote nodes on the bus see it, but co-hosted sibling nodes do not.

Without a fix, this affects every protocol:

ProtocolImpact without sibling dispatch
Event TransportNode A produces an event; co-hosted Node B that consumes it never sees the PCER.
DatagramsNode A sends a datagram addressed to Node B; Node B never receives it.
Train ControlA local throttle node sends speed commands to a co-hosted train node; the train never responds.
Verify Node IDGlobal Verify from Node A should elicit a response from Node B; without loopback it does not.
Init CompleteNode A's Init Complete must be seen by Node B to update its listener alias table; without loopback it is missed.

21.3 Zero-Copy Sequential Sibling Dispatch

The library solves the loopback problem with a zero-copy, zero-FIFO sequential dispatch mechanism built into the main dispatcher run loop.

Core idea

After a message is sent to the wire, the dispatcher reuses the outgoing message buffer as the incoming message for a second pass through all sibling nodes. No buffer is allocated and no FIFO push occurs. One sibling is visited per _run() call.

21.4 Dispatch Phases

Phase 1 — Incoming message fan-out (main state machine)

Every incoming message from the FIFO is already dispatched to all nodes via the enumerate-first / enumerate-next loop (Ch 8 §8.5). This handles messages arriving from the wire. Global messages reach every node; addressed messages are filtered by does_node_process_msg().

Phase 2 — Outgoing sibling dispatch (main state machine)

After a node sends an outgoing message to the wire, the dispatcher automatically shows it to every other local node. The run loop priority 2 sub-steps handle this (see Ch 8 §8.2). Self-skip is enforced: the originating node never processes its own outgoing message because state.loopback = true is set and does_node_process_msg() rejects it.

Phase 3 — Login sibling notification (login state machine)

The login state machine calls OpenLcbMainStatemachine_send_with_sibling_dispatch() instead of the raw transport send when transmitting Init Complete and event announcement messages. This wrapper sends the message to the wire and queues it for sibling dispatch via the Path B pending slot, ensuring that login-time messages also reach co-hosted nodes.

21.5 Chain Dispatch (depth > 1)

Sibling dispatch can be recursive: Node A's Init Complete triggers Node B's listener table update, which may itself cause Node B to send an Init Complete. Node B's Init Complete must then be shown to Node A and Node C. The sibling response queue handles this cleanly:

  1. When a sibling sends a response during dispatch, the response is pushed to _sibling_response_queue.
  2. After the current sibling dispatch cycle completes, Priority 2.5 pops the queued response and starts a new sibling dispatch cycle for it.
  3. This continues until the queue is empty.

21.6 Per-Node Alias Allocation

Each virtual node goes through the full CAN login sequence independently via round-robin processing in the CAN main state machine. The login state machine iterates through all nodes not yet in RUNSTATE_RUN, running one state transition per node per main loop iteration. All nodes log in concurrently, each generating its own alias from its own Node ID seed.

21.7 Single-Node Fast Path

On single-node configurations, _sibling_dispatch_begin() checks openlcb_node_get_count() <= 1 and returns immediately. There is no overhead from sibling dispatch when only one node is configured.

21.8 Timing and Buffer Pressure

Each sibling is visited in one _run() call. The work per call is approximately 10 µs of CPU, so 50 siblings ≈ 500 µs — less than one CAN frame time at 125 kbps (~1 ms). The CAN RX ISR continues filling its software FIFO independently, so no frames are lost during sibling dispatch.

No dynamic allocation occurs during sibling dispatch. All buffers are statically allocated at startup.
